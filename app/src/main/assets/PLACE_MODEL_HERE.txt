NsfwClassifierFactory looks for nsfw.onnx first, then falls back to
nsfw.tflite, then to StubNsfwClassifier (always scores 0, never blocks) if
neither is present. As of this commit, nsfw.onnx is in place
(AdamCodd/vit-base-nsfw-detector, model_fp16.onnx variant - see
OnnxNsfwClassifier.kt for the NNAPI/CPU session setup and ViTPreprocessor.kt
for the 384x384/mean-0.5/std-0.5 preprocessing it expects).

--- Legacy TFLite path (nsfw.tflite) ---

Expected contract (see detect/TFLiteNsfwClassifier.kt):
  - Single image input, any HxW in the 128-160px range, NHWC layout.
  - Either float32 (normalized 0..1) or fully-integer-quantized
    (uint8/int8 with the standard TFLite scale/zero-point metadata) input.
  - Single-value output tensor: one NSFW probability in [0,1] (float32
    output, or quantized uint8/int8 dequantized via scale/zero-point).

TFLiteNsfwClassifier reads input size and dtypes off the model itself, so
as long as the shapes/types above hold, no code changes are needed beyond
dropping the file here. NsfwClassifierFactory falls back to
StubNsfwClassifier (always scores 0, never blocks) whenever this file is
absent - which is the state this repo ships in.

This file itself (PLACE_MODEL_HERE.txt) is harmless to leave in the APK;
delete it once nsfw.tflite is in place if you'd like.

Recommended source model: GantMan/nsfw_model (MIT licensed), MobileNetV2
variant - https://github.com/GantMan/nsfw_model. It's a 5-class softmax
(drawings/hentai/neutral/porn/sexy), not a single sigmoid, but
TFLiteNsfwClassifier already handles that (see its unsafeClassIndices
param - defaults to summing hentai+porn+sexy). See tools/convert_nsfw_model.py
for the exact conversion/quantization script and preprocessing details.
